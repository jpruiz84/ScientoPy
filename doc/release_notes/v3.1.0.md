# ScientoPy v3.1.0 — Release Notes

## Highlights

ScientoPy 3.1.0 focuses on startup performance, GUI responsiveness, and Windows code signing. The application now shows a splash screen within milliseconds of launch, heavy modules are deferred until first use, graph appearance is independently configurable, and the Windows executable is signed via the SignPath Foundation OSS program.

---

## Performance

- **Fast startup with lazy imports:** PySide6, matplotlib, polars, numpy, scipy, and PIL are no longer imported at module load time. They are deferred to first use, cutting cold-start time significantly and warm-start time from ~5 s to under 1 s.
- **Instant splash screen:** A lightweight tkinter splash ("ScientoPy — Loading...") appears in a background thread before any heavy module loads, giving immediate visual feedback.
- **Parallel CSV loading with threads:** The `load_folder()` I/O stage now uses `ThreadPoolExecutor` instead of `ProcessPoolExecutor`. Polars CSV parsing releases the GIL (Rust), so threads provide real parallelism without the fork-safety issues that caused hangs on repeated preprocess runs in the GUI.
- **Parallel dedup key computation with threads:** Same switch in `compute_dedup_keys_parallel()` for the same reason.

## Bug Fixes

- **Fix graph disappearing after reducing topic count:** Analyzing with 200 topics (no graph shown, as expected) then reducing back to 10 showed no graph. Root cause: `plotResults()` was mutating `self.noPlot = True` permanently on the first large run. Fixed by removing the mutation — the function now just returns early without modifying state.
- **Fix preprocess hanging on second GUI run:** `ProcessPoolExecutor` with `fork()` inherited matplotlib and Qt lock state from the GUI process, causing deadlocks on the second preprocess. Fixed by switching to `ThreadPoolExecutor`.
- **Fix "Not Responding" dialog during word cloud generation:** `wc.generate_from_frequencies()` (CPU-bound, can take several seconds for 1000 topics) was blocking the Qt main thread. Fixed by pre-computing the word cloud in a worker thread behind a `ProgressDialog`, then handing the result to `plotResults()` for display on the main thread.
- **Fix Windows CI failure (Unicode encoding):** `PreProcessClass._announce_step()` printed Unicode box-drawing characters (`─`, `═`, `✓`, `…`, `→`) which the Windows cp1252 console codec cannot encode. Replaced all such characters in `print()` calls with ASCII equivalents (`-`, `=`, `OK`, `...`, `->`) across `PreProcessClass.py` and `exportPapers.py`.

## New Features

- **Graph appearance menu (View → Graph appearance):** Independent control for matplotlib figure theming with three modes:
  - **Light** — white background, dark axes (default; consistent across all OS themes)
  - **Dark** — dark background, light axes
  - **Same as UI** — follows the application's Light/Dark/System appearance setting
- **Graph appearance applies to all graphs:** Analysis graphs, preprocess graphs, and word clouds all respect the selected mode.

## Code Signing (Windows)

- **SignPath OSS Foundation integration:** The Windows executable is now submitted for Authenticode code signing via the [SignPath Foundation](https://signpath.org) OSS program as part of the GitHub Actions release pipeline.
- **Artifact configuration:** `.signpath/artifact-configuration.xml` defines which files inside the release ZIP are signed (`ScientoPyGui.exe`, `*.dll`, `*.pyd`).
- The `sign-windows` CI job runs automatically on every `v*` tag push, after the build job completes.

## Infrastructure

- **`paperIO.py` — thread-safe parallel I/O:** Replaced both `ProcessPoolExecutor` usages with `ThreadPoolExecutor` for GUI compatibility.
- **Splash screen dismissed before window show:** `_splash_done.set()` is called just before `window.show()` so the splash never outlasts the main window.
